## 3. Issues Found

### Issue 1 — RACE DETAIL PAGES MISSING FROM SITEMAP (High)

**Severity:** High

`/races/[slug]` is a fully-rendered, canonical page with Event JSON-LD and unique metadata per race. It is not included in the sitemap.

The sitemap fetches all races (via `fetchAllRaces()`) purely to compute which `races/[scope]/[city]` combinations pass the quality gate — but never emits individual race detail URLs.

**Impact:** Google will still discover race detail pages via internal links (race cards on `/races` link to `/races/[slug]`), but omitting them from the sitemap delays indexing and deprives Googlebot of explicit `lastmod` signals.

**Fix:** After building the race list in `sitemap.ts`, emit one entry per race:

```ts
for (const race of races) {
  const slug = race.slug || race.id;
  staticRoutes.push({
    url: `${SITE}/races/${slug}`,
    lastModified: race.updatedAt ? new Date(race.updatedAt) : new Date(race.startTime),
    changeFrequency: 'weekly',
    priority: 0.7,
  });
}
```

Note: the `ApiEvent` type in `sitemap.ts` is imported from `@/app/races/page` and includes `slug`, `id`, and `startTime`, so this is a straightforward addition.

### Issue 2 — UNPUBLISHED CLUBS MAY APPEAR IN SITEMAP (High)

**Severity:** High

`/src/app/clubs/[slug]/page.tsx` calls `notFound()` when `!club.publishedAt` and sets `robots: { index: false }` in `generateMetadata`. However, `sitemap.ts` emits a `/clubs/[slug]` entry for every slug returned by `GET /api/v1/clubs`.

If the clubs list endpoint returns unpublished clubs (or clubs not yet fully set up), their URLs will appear in the sitemap but return either 404 or a noindexed page — both are validation failures.

**Fix:** Either filter the clubs list API to published-only before emitting sitemap entries, or add a `publishedAt` field to the `ListedClub` type in `sitemap.ts` and gate on it:

```ts
// In sitemap.ts ListedClub type:
type ListedClub = { slug: string; name: string; city: string; updatedAt: string | null; publishedAt: string | null };

// In the emit loop:
for (const c of clubs) {
  if (!c.publishedAt) continue;  // skip unpublished
  staticRoutes.push({ url: `${SITE}/clubs/${c.slug}`, ... });
}
```

Confirm whether the API already filters to published clubs. If yes, this is a documentation gap only.

### Issue 3 — LASTMOD IS GENERATION TIME, NOT CONTENT DATE (Low)

**Severity:** Low

All static routes except `/privacy`, `/terms`, and `/support` use `new Date()` as `lastmod`. Since the sitemap revalidates every hour, Googlebot sees a constantly-updated `lastmod` on every page — including pages whose content hasn't changed. This trains Googlebot to treat `lastmod` as unreliable, reducing its utility as a crawl-priority signal.

Static pages like `/`, `/runners`, `/coaches`, `/workout-plan`, and `/club-pitch-deck` change infrequently and should use a fixed date anchored to the most recent deployment or content change.

**Fix (low effort):**

```ts
{ url: `${SITE}/`,            lastModified: new Date('2026-04-15'), changeFrequency: 'weekly',  priority: 1.0 },
{ url: `${SITE}/races`,       lastModified: new Date(),             changeFrequency: 'daily',   priority: 0.9 }, // OK: data changes daily
{ url: `${SITE}/clubs`,       lastModified: new Date(),             changeFrequency: 'daily',   priority: 0.9 }, // OK: data changes daily
{ url: `${SITE}/runners`,     lastModified: new Date('2026-04-15'), changeFrequency: 'weekly',  priority: 0.85 },
{ url: `${SITE}/coaches`,     lastModified: new Date('2026-04-15'), changeFrequency: 'weekly',  priority: 0.85 },
{ url: `${SITE}/workout-plan',lastModified: new Date('2026-04-01'), changeFrequency: 'monthly', priority: 0.7 },
```

### Issue 4 — /whitepaper AND /club-pitch-deck ASYMMETRY (Info)

**Severity:** Info

`/whitepaper` has a page file at `src/app/whitepaper/page.tsx` but is not in the sitemap. `/club-pitch-deck` is in the sitemap at `priority: 0.7`.

If `/whitepaper` is a public-facing content page it should be included. If it is gated or draft, add `robots: noindex` to its `generateMetadata`.

---

### Issue 5 — REDIRECT VANITY URLS IN SITEMAP RISK (Low)

**Severity:** Low / Info

`/ggm26-ig` and `/mbe-ig` are `permanent: false` (302) redirects in `next.config.ts`. They are correctly absent from the sitemap. The targets (`/races/7th_edition_gurugram_grand_half_marathon_2026_36780`, `/clubs/mberunclub`) are real canonical URLs — the race detail is missing from the sitemap entirely (see issue #1), and the club is only present if the clubs API returns it.

## 5. Recommended Fixes — Priority Order

1. **Add race detail pages to sitemap** (High) — `src/app/sitemap.ts`, emit `/races/${slug}` for each race returned by `fetchAllRaces()`.
2. **Gate sitemap club entries on `publishedAt`** (High) — either confirm API filters to published clubs, or add the `publishedAt` check to the emit loop.
3. **Verify `robots.txt` declares `Sitemap:` directive** (High) — run `curl https://www.endorfin.run/robots.txt` and confirm `Sitemap: https://www.endorfin.run/sitemap.xml` is present.
4. **Fix `lastmod` on truly-static routes** (Low) — use fixed dates for `/`, `/runners`, `/coaches`, `/workout-plan`, `/club-pitch-deck`.
5. **Resolve `/whitepaper` status** (Info) — either add to sitemap + add footer link, or set `robots: { index: false }`.
6. **Add internal link to `/club-pitch-deck`** (Info) — it is in the sitemap but has no discoverable inbound link in the rendered site nav or footer.
7. **Remove `priority` and `changeFrequency`** (Optional/Info) — Google ignores both fields. Removing them reduces XML payload size and avoids future confusion.
